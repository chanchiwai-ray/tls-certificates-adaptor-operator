# Research References

## Summary

The **tls-certificate-adaptor-operator** charm bridges two fundamentally different TLS certificate interface designs in the Juju ecosystem. On the **old side**, Charmed OpenStack charms (Yoga and earlier) use the reactive `interface-tls-certificates` ([ref 1][R1]), where the CA provider (e.g., charm-vault from OpenStack) generates **both the private key and the certificate** and returns both to the requirer via plain relation data. On the **new side**, the `tls-certificates-interface` v4 (now superseded by `charmlibs.interfaces.tls_certificates`) ([ref 2][R2], [ref 3][R3]) requires the **requirer to generate its own private key**, send only a CSR, and receive back only the signed certificate. Private keys never leave the unit in the new interface.

The adaptor acts as a **provider on the old interface side** (accepting `common_name` + `sans` requests from OpenStack charms) and as a **requirer on the new interface side** (sending CSRs to vault-k8s and receiving signed certificates). Since the old interface contracts require the adaptor to return a private key alongside the certificate, the adaptor must **generate and manage private keys on behalf of old-style requesters**. These keys must be written into Juju relation data (not Juju Secrets), because the old reactive charms have no knowledge of Juju Secrets. The new interface internally stores the adaptor's own upstream private keys in Juju Secrets (Juju >= 3.0 required).

The reactive `interface-tls-certificates` ([ref 1][R1]) defines flags (`{endpoint}.certs.requested`, `{endpoint}.server.certs.requested`, etc.) to trigger certificate issuance on the provider side. The new `tls-certificates-interface` v4 ([ref 2][R2]) defines events (`certificate_available`) and uses `_RequirerData` / `_ProviderApplicationData` Pydantic models to read/write relation data, storing a list of CSRs in the requirer databag and a list of issued certificates in the provider app databag.

There is no existing upstream adaptor charm. The charm is a new piece of infrastructure that must be implemented from scratch, following the standard charm design patterns ([ref 8][R8]).

## Key Findings

- **Old interface (reactive)**: `CertificateRequest` objects carry `common_name`, `sans`, and `cert_type` (server/client/application/intermediate). The provider responds with a `Certificate(cert_type, common_name, cert, key)` stored at keys like `{munged_unit_name}.processed_requests` in the provider's unit relation data. **Both cert and private key are in relation data.** ([ref 1][R1], [ref 5][R5])

- **Old interface — key types supported**: server, client, application (shared across all units), intermediate CA. The adaptor initially needs to support at minimum the `server` type, which is what OpenStack API services (keystone, nova, cinder, etc.) use. ([ref 6][R6])

- **New interface (v4 / charmlibs)**: Requirer places `{"certificate_signing_requests": [{"certificate_signing_request": PEM, "ca": bool}]}` in its unit/app databag. Provider places `{"certificates": [{"certificate_signing_request": PEM, "certificate": PEM, "ca": PEM, "chain": [PEM...], "revoked": bool}]}` in its app databag. Private keys never appear in relation data. ([ref 2][R2], [ref 3][R3])

- **Juju Secrets and private keys**: Juju Secrets support cross-application sharing (owner creates secret, grants it to observer app, passes secret ID via relation data). However, old reactive charms expect the private key as a plaintext string in relation data. **Juju Secrets cannot replace relation data for the old→requirer direction**; the old charm must be updated to read secrets, which defeats the purpose of the adaptor. The adaptor's own internal key management (for the new interface leg) can and should use Juju Secrets. ([ref 7][R7])

- **Security implication**: Private keys generated by the adaptor for old-style requesters will be stored in the Juju controller database (via unit relation data). This is a known, accepted limitation of the old interface design. The adaptor should document this prominently and be positioned as a **migration tool** to enable gradual upgrade of OpenStack charms, not a permanent solution.

- **Certificate lifecycle**: Old charms react to `{endpoint}.certs.changed` when a certificate or CA is updated. When vault-k8s rotates a certificate, the adaptor receives a `certificate_available` event (new interface), must regenerate a new private key and update the old relation data. The adaptor must also track which CSR corresponds to which old-interface request so it can route the response back correctly.

- **Vault comparison (OpenStack vs k8s)**: `charm-vault` (OpenStack) uses a reactive framework and generates cert+key from CN+SANs via Vault's PKI secrets engine. `vault-k8s` (canonical) uses the ops framework, accepts CSRs, and returns only signed certificates. ([ref 4][R4], [ref 9][R9])

- **Library deprecation**: `charms.tls_certificates_interface.v4.tls_certificates` is deprecated; the preferred library is `charmlibs.interfaces.tls_certificates` (pip: `charmlibs-interfaces-tls-certificates`). For the new interface leg, the adaptor should use `charmlibs`. ([ref 2][R2])

- **Juju version requirement**: The new interface requires Juju >= 3.0 for Juju Secrets. The adaptor charm must document this requirement. Charmed OpenStack charms using the reactive interface can coexist with Juju 3.x.

- **No existing adaptor charm found** in the community. This is a greenfield implementation.

```
OpenStack charm          Adaptor                   vault-k8s
(keystone, nova...)
    |                       |                          |
    |-- common_name, sans -->|                          |
    |   (old interface)      |-- CSR (adaptor's key) -->|
    |                       |<-- cert + CA + chain -----|
    |<-- cert + key + CA ----|   (new interface)
    |   (old interface)
```
